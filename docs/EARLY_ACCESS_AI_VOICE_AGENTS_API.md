# Early Access API — AI Agents & Voice Agents

This document defines the backend endpoints required by:

- `src/pages/AIAgents.tsx`
- `src/pages/VoiceAgents.tsx`

It covers:

- **Early access waitlist** submission
- **Mandatory KYC file upload**
- **AI Agents builder** endpoints (create/save/simulate/deploy)

## Conventions

- **Base URL**: `https://sms.mifumolabs.com` (production) or your local API host
- **Auth**:
  - Waitlist endpoints support **anonymous** submissions, but may also accept JWT auth.
  - If the frontend has a token, it will send `Authorization: Bearer <access_token>`.
- **Response shape**: recommended unified format

```json
{
  "success": true,
  "message": "Human readable message",
  "data": {}
}
```

- **Error shape** (recommended)

```json
{
  "success": false,
  "message": "Validation error",
  "errors": {
    "field_name": ["Reason 1", "Reason 2"]
  }
}
```

## 1) Early Access Waitlist + KYC Upload (Required)

Both pages submit a **multipart/form-data** payload including a required KYC file.

### 1.1 POST `/api/early-access/ai-agents/waitlist/`

**Used by**: `AIAgents.tsx` “Join the waitlist” form.

**Auth**: Optional `Bearer` token.

**Content-Type**: `multipart/form-data`

#### Request (multipart fields)

- `full_name` (string, required)
- `email` (string, required)
- `phone` (string, optional, recommended E.164 e.g. `+255689726060`)
- `kyc_file` (file, required)
  - Allowed: PDF/JPG/JPEG/PNG
  - Recommended max size: 10–20MB

#### Response (201 Created)

```json
{
  "success": true,
  "message": "Waitlist request received",
  "data": {
    "id": "wl_ai_01J...",
    "product": "ai_agents",
    "full_name": "John Doe",
    "email": "john@company.com",
    "phone": "+255689726060",
    "kyc_file_url": "https://sms.mifumolabs.com/media/kyc/wl_ai_01J....pdf",
    "status": "pending",
    "created_at": "2026-03-27T12:01:22Z"
  }
}
```

#### Response (400 Validation error)

```json
{
  "success": false,
  "message": "Validation error",
  "errors": {
    "email": ["Invalid email"],
    "kyc_file": ["This field is required"]
  }
}
```

#### Curl test

```bash
curl -X POST "https://sms.mifumolabs.com/api/early-access/ai-agents/waitlist/" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -F "full_name=John Doe" \
  -F "email=john@company.com" \
  -F "phone=+255689726060" \
  -F "kyc_file=@./kyc.pdf"
```

### 1.3 Backend storage + security requirements (KYC)

Recommended controls:

- **Virus/malware scanning** (if possible)
- **Private file storage** (do not expose raw directory listing)
- **Signed URLs** for retrieval (time-limited)
- **Audit log**: who uploaded, when, IP/user agent, and the associated request
- **PII/KYC retention policy**: expiry + delete flow

## 2) AI Agents Builder API (Create → Save Flow → Simulate → Deploy)

The `AIAgents.tsx` builder produces two key payloads:

- **Create payload** (onboarding): `name`, `business`, `industry`, `language`, `tone`, `channel`, `intent`
- **Flow contract** (runtime JSON): generated by the builder and shown in “Flow contract”

### 2.1 POST `/api/chatbots`

Create an AI agent profile (draft).

**Auth**: Required (recommended) — this is a product feature, not public marketing.

#### Request JSON

```json
{
  "name": "Kuza Sales Agent",
  "business": "Mifumo Labs",
  "industry": "Retail",
  "language": "mixed",
  "tone": "sales",
  "channel": "sms",
  "intent": "Help customers buy faster, check pricing, and request a human sales follow-up."
}
```

#### Response (201 Created)

```json
{
  "success": true,
  "message": "Chatbot created",
  "data": {
    "chatbot_id": "cb_01J...",
    "status": "draft"
  }
}
```

---

### 2.2 POST `/api/chatbots/{chatbot_id}/flow`

Save the structured flow contract.

#### Request JSON

```json
{
  "flow": {
    "START": {
      "message": "Karibu...",
      "options": {
        "View packages": "SMS_PACKAGES",
        "Talk to sales": "SALES_HANDOFF"
      }
    }
  }
}
```

#### Response (200 OK)

```json
{
  "success": true,
  "message": "Flow saved",
  "data": {
    "chatbot_id": "cb_01J...",
    "version": 3
  }
}
```

---

### 2.3 POST `/api/chatbots/{chatbot_id}/simulate`

Run a test session from `START` and return the next state transitions.

#### Request JSON (minimum)

```json
{
  "start_state": "START",
  "input": "View packages"
}
```

#### Response (200 OK)

```json
{
  "success": true,
  "data": {
    "current_state": "START",
    "output_message": "We have Starter, Growth, and Scale packages...",
    "options": ["Starter", "Growth", "Back to menu"],
    "next_state": "SMS_PACKAGES"
  }
}
```

---

### 2.4 POST `/api/chatbots/{chatbot_id}/deploy`

Publish the agent to production.

#### Request JSON

```json
{
  "environment": "production"
}
```

#### Response (200 OK)

```json
{
  "success": true,
  "message": "Deployed",
  "data": {
    "chatbot_id": "cb_01J...",
    "status": "active",
    "deployed_at": "2026-03-27T12:45:00Z"
  }
}
```

---

### 2.5 POST `/api/chatbots/{chatbot_id}/channels/sms`

Attach SMS channel configuration.

#### Request JSON (example)

```json
{
  "sender_id": "MIFUMOSMS",
  "inbound_enabled": true,
  "outbound_enabled": true,
  "webhook_secret": "optional"
}
```

---

### 2.6 POST `/api/chatbots/{chatbot_id}/webhooks`

Attach external automation handlers (CRM, lead capture, support ticket, etc.)

#### Request JSON (example)

```json
{
  "name": "sales-lead",
  "url": "https://example.com/webhooks/sales-lead",
  "secret": "shared-secret",
  "enabled": true
}
```

## 3) Voice Agents (Suggested endpoints for “real data”)

`VoiceAgents.tsx` is currently a marketing/early access surface. To make all sections server-driven, implement:

- **GET** `/api/early-access/voice-agents/features/` (optional)
  - Returns the list of “Feature Preview” cards
- **GET** `/api/early-access/voice-agents/status/` (optional)
  - Returns current program status: `private_beta`, capacity, etc.

Example response:

```json
{
  "success": true,
  "data": {
    "program": "voice_agents",
    "status": "private_beta",
    "features": [
      { "key": "voice_dashboard", "title": "Voice Dashboard", "desc": "..." }
    ]
  }
}
```

## 4) Notes about `/tutorial` route conflict (deployment)

If your server serves `/tutorial` as a real directory, do **not** use `/tutorial` as a SPA route.
Use `/watch-tutorial` (SPA route) and keep `/tutorial/<file>` only for static assets if needed.
